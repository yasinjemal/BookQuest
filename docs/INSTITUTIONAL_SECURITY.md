# BookQuest institutional security and data-flow brief

Last updated: 2026-07-13

This document describes the implemented pilot architecture. It is not a
certification, legal opinion or claim that BookQuest satisfies every regulatory
framework. Contractual commitments must match the selected partner, deployment
configuration and completed external assessment.

## Data flow and system boundaries

1. A user connects to the BookQuest web application over HTTPS. Vercel runs the
   application and API. Authentication cookies are HTTP-only, same-site and secure
   in production.
2. PostgreSQL records are stored in Neon. The current production database is in an
   EU Central AWS region. Application runtime residency is not claimed to be pinned
   until a partner contract and Vercel region configuration are verified.
3. Uploaded source text may be sent to the configured generation provider when AI
   generation is enabled. BookQuest stores the source/version provenance and exact
   generated course version. Manual mode avoids the generation API.
4. Resend receives recipient email addresses and transactional message content when
   email delivery is configured. Local development uses preview delivery.
5. Flutterwave receives checkout/customer fields only when a user starts a payment.
6. Audit packs are generated in the application process from authorized,
   tenant-scoped PostgreSQL queries. They are not public objects. The caller receives
   the generated bytes and the database retains a manifest and artifact hash.

## Subprocessor register for pilot review

| Provider | Purpose | Typical data | Required pilot check |
|---|---|---|---|
| Vercel | Web hosting and serverless runtime | requests, session cookie, application responses | contracted region, logs, DPA and deletion |
| Neon | PostgreSQL hosting, backup and point-in-time recovery | accounts, content, evidence, audit metadata | EU Central project, retention, PITR and access list |
| Anthropic | Optional course generation | source excerpts and generation prompts | model/data-retention terms and approved source classes |
| Resend | Transactional email | email address and message | DPA, delivery retention and sender domain |
| Flutterwave | Optional payment processing | checkout identity and payment fields | merchant terms, geography and financial retention |

No provider should be represented as approved by a partner until its security,
privacy and procurement review is complete.

## Identity and access controls

- Space authorization is deny-by-default and tenant scoped. Platform operators do
  not gain implicit tenant access through an application admin role.
- Organization roles separate owner, administrator, creator, reviewer, manager,
  learner and read-only auditor capabilities.
- TOTP authenticator secrets are AES-256-GCM encrypted with a deployment secret.
  Login challenges and recovery codes are one-time and stored only as digests.
- Organization policy versions set password length, session lifetime, MFA-required
  roles, retention and legal-hold behavior. A policy cannot require MFA from an
  unenrolled member. Publishing a policy revokes current organization sessions.
- OIDC/SAML configuration is represented but remains disabled until a real pilot
  selects and tests one provider. SCIM remains volume-triggered.

## Evidence integrity and privacy

- Published courses, completion rules, assignment versions and evidence events are
  immutable. Lifecycle projections change only alongside append-only events.
- Completion binds the exact course, assignment and rule versions to evidence IDs
  and a canonical SHA-256 hash.
- Public credential verification requires a high-entropy private token. Display
  codes cannot enumerate credentials. Expiry and revocation apply immediately.
- Legal holds require a scope and reason and block Space deletion scheduling.
- Account export excludes passwords, sessions, MFA secrets and security tokens.

## Continuity, recovery and incident response

- Logical backup restoration and ledger reconciliation run in CI. Production PITR
  and reliability evidence is recorded under `docs/evidence/` with measured RPO/RTO.
- Migration execution is transactional and protected by a transaction-scoped
  advisory lock compatible with pooled PostgreSQL connections.
- The incident sequence is: contain credentials/sessions, preserve tenant-scoped
  evidence, assess affected versions and subjects, notify the responsible owner,
  recover from the verified restore point, reconcile ledgers and publish a factual
  post-incident record. Notification timing must follow the applicable contract and
  law; BookQuest does not claim one universal deadline.
- Recovery exercises must be repeated after material storage/topology changes and
  before a partner production launch.

## Pilot acceptance checklist

Organization owners use `/spaces/{space-id}/pilot` to version the baseline and
success criteria, append pseudonymous observations, bind stakeholder/assessor
decisions to exact evidence and view the remaining governed gates. The application
refuses closure while the required database evidence or accepted decisions are
missing. This control is an evidence register, not a substitute for real partner or
independent-assessor work.

- Confirm contracted runtime and database regions.
- Approve the final subprocessor list and data classes sent to optional providers.
- Test the partner identity provider and recovery/lockout process.
- Complete an independent penetration test and close material findings.
- Complete the full WCAG 2.2 AA journey audit.
- Run backup/restore and incident tabletop exercises with named owners.
- Have the responsible stakeholder accept one scoped audit pack for its stated use.
