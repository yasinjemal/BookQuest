# BookQuest Product Blueprint

**Status:** living product and delivery plan  
**Last updated:** 12 July 2026

## Executive decision

BookQuest will build two reinforcing assets:

1. A learning evidence base connecting concepts, question versions, answers,
   explanations, and later learning outcomes.
2. Trusted organizational records showing who learned what, from which document
   version, under which completion rule, and with what evidence.

The evidence foundation starts immediately because historical answers cannot be
backfilled. The first commercial product remains the Compliance Training Engine.

The delivery order is therefore:

1. Thin Learning Evidence Ledger
2. Compliance Training Engine
3. Skill Passport and verification
4. Learning Genome products as the dataset becomes statistically useful
5. WhatsApp delivery after the learning and commercial loops work

This is a sequence, not four disconnected products. Every stage should reuse the
same versioned content, concept identities, answer evidence, mastery projections,
and access controls.

## Product thesis and moat

Course-generation algorithms can be copied. A trustworthy, permissioned history
of how real learners interact with document-derived concepts cannot be copied or
reconstructed later.

BookQuest's compounding advantage is the relationship between:

- the source document and its published version;
- the concept being taught;
- the exact version of a question and explanation;
- the learner's normalized response, timing, hints, and retries;
- mastery before and after the interaction; and
- later retention or successful application.

The data is only an advantage if it is complete, well-versioned, legally usable,
tenant-safe, and difficult to game. Raw volume without those properties is noise
and liability.

## Strategic pillars

### 1. The Learning Genome

The Learning Genome is a versioned concept and evidence system that improves
learning decisions as eligible answers accumulate.

#### Capabilities

- Course-scoped concept identities and later confidence-scored cross-course links
- Prerequisite relationships between concepts
- Learner mastery estimates with algorithm and taxonomy versions
- Question difficulty, discrimination, ambiguity, and retirement signals
- Explanation-effectiveness measurement
- Placement tests that skip already-mastered material
- Adaptive learning and review paths
- Recognition of equivalent skills learned through different documents

#### Development stages

1. Capture every answer exactly once.
2. Measure simple, interpretable question statistics.
3. Flag poor questions for human review.
4. Produce course-level placement recommendations.
5. Infer prerequisite edges with confidence scores.
6. Match concepts across courses with human correction.
7. Add adaptive paths and controlled explanation experiments.
8. Consider formal item-response models only after representative sample sizes
   are available per item and cohort.

Concept labels generated by AI are not automatically global truth. BookQuest will
begin with course-scoped concepts. Global matching is a separate, reversible
process with provenance and confidence.

Private organizational content remains tenant-scoped unless its owner explicitly
permits a defined form of aggregation. Public and consented data must remain
distinguishable throughout the pipeline.

### 2. Skill Passport and Employer Verification

The Skill Passport is a learner-controlled profile of evidence-backed competency
claims. It is stronger than a completion badge because it exposes the quality,
quantity, recency, and conditions of the underlying evidence.

#### Claim contents

- Competency ID, label, and taxonomy version
- Mastery estimate and confidence interval or confidence band
- Evidence volume, first/last evidence dates, and recency status
- Source courses and content versions
- Assessment conditions and delivery channel
- Scoring and mastery algorithm versions
- Credential ID, issuer, issue date, expiry, and revocation state
- Verification URL or API representation

#### Learner controls

- Private by default
- Explicit selection of claims to share
- Expiring and revocable verification links
- Access history
- Consent withdrawal
- Correction and dispute workflow
- Export in a portable format

#### Employer API

The initial API is read-only and scoped. It verifies a claim and its evidence; it
does not rank candidates, infer employability, or make hiring recommendations.

Possible endpoints:

- Verify a credential by opaque ID
- Read claims covered by a learner-granted token
- Check expiry or revocation status
- Retrieve summarized evidence and version metadata

Consequential employment decisions must remain human decisions. Confidence,
sample size, expiry, and limitations must be visible rather than hidden behind a
single score.

### 3. Compliance Training Engine

The Compliance Training Engine turns an organization's controlled documents into
assigned, auditable learning.

Core workflow:

**Upload document → generate draft → review → publish immutable version → assign
learners → collect evidence → apply completion rule → export audit pack**

#### Compliance MVP

- Organizations with strict tenant isolation
- Owner, administrator, manager, learner, and auditor roles
- Invitations, teams, and membership lifecycle
- Organization-owned document library
- Draft review and approval before publication
- Immutable published course versions
- Assignments, due dates, reminders, and reassignment
- Configurable completion rules and pass thresholds
- Learner acknowledgement or attestation
- Evidence-linked certificates
- Certificate expiry and revocation
- CSV and PDF audit-ready exports
- Append-only administrative audit log
- Course-version and assignment history

#### Audit pack

An export should identify:

- organization and reporting scope;
- learner and assignment;
- source document and published content version;
- assignment, start, completion, and expiry dates;
- assessment attempts and evidence IDs;
- completion rule and score calculation version;
- attestation, certificate, and revocation state; and
- report generator, generation time, and report version.

BookQuest should promise **audit-ready evidence**, not universal regulatory
compliance. Requirements vary by jurisdiction and industry and require local
legal review before specialized claims are marketed.

#### Commercial hypothesis

The initial offer is recurring per-active-seat software with an organization base
fee. Paid pilots should focus on one or two document-heavy sectors and validate:

- time from document upload to assigned training;
- administrative time saved;
- audit-pack usefulness;
- completion improvements; and
- renewal and seat expansion.

Pricing is a hypothesis until pilot interviews and willingness-to-pay tests are
complete.

### 4. WhatsApp Learning Delivery

WhatsApp is a distribution channel, not a separate learning engine. The same
lesson, evidence, mastery, assignment, and consent services must support web and
chat delivery.

#### First pilot

- Explicit account linking and opt-in
- Short lesson cards delivered in sequence
- Numbered or short-text answers
- Answer evidence recorded through the same server path
- Assignment reminders and resume prompts
- Progress synchronized with the web app
- STOP/help handling and consent history
- Delivery status and messaging-cost analytics
- Secure, short-lived web links for sensitive or complex actions

#### Channel constraints

- Inbound messages and delivery webhooks must be idempotent.
- Phone numbers stay in the identity boundary, not the learning ledger.
- Chat should not expose proprietary course content to the wrong account.
- Rich reports, document approval, and complex administration remain web flows.
- Learning outcomes and completion should be compared with web delivery, not
  assumed equivalent.

## Shared platform foundation

### Learning Evidence Ledger

Every submitted answer is an immutable event. Current-state tables such as lesson
progress, review schedules, and concept mastery remain fast projections, not the
only evidence source.

Minimum evidence includes:

- idempotent event ID;
- occurrence and server-receipt timestamps;
- pseudonymous learner key;
- organization, enrollment, and assignment context when available;
- course and content version;
- lesson, stable question identity, question version, and concept identity;
- generator model and prompt versions when known;
- normalized response evidence, correctness, skip, timing, attempt, and hint data;
- mastery before and after the answer;
- mastery algorithm version;
- delivery channel, privacy scope, consent version, and retention class.

Names, email addresses, phone numbers, and arbitrary raw learner-entered text do
not belong in the ledger. Identity mappings are stored separately so erasure or
key destruction can anonymize evidence without corrupting aggregate statistics.

### Versioning

The following changes produce new versions rather than overwrites:

- published course content;
- questions or answer keys;
- explanations;
- concept taxonomy mappings;
- generator model or prompt;
- mastery and scoring algorithms;
- completion rules; and
- credential formats.

Historical evidence keeps its original version references.

### Derived projections

- Current learner-course progress
- Current learner-concept mastery
- Review schedule
- Question calibration statistics
- Assignment completion
- Credential claims
- Organization reports

Each important projection should eventually be reproducible from immutable events
plus its versioned rules.

## Delivery roadmap

### Phase 0 — Evidence foundation (in progress)

Build:

- Stable question and question-version identities
- Course content versions and generator provenance
- Pseudonymous learner identities
- Append-only, idempotent answer events
- Server-authoritative grading
- Atomic mastery projection updates
- Durable browser outbox for offline/retry delivery
- Persisted standard and fresh practice sessions
- Admin ledger-health summary
- Runtime validation and automated tests

Release gates:

- Each accepted answer produces exactly one event.
- A replay cannot increment mastery or XP twice.
- The server derives correctness from its question version.
- Lesson, practice, and review answers use the same evidence path.
- Fresh generated questions exist before they can be answered.
- Failed network delivery remains queued under the original event ID.
- No direct personal identifier or arbitrary raw fill-in response is stored.
- Database integrity, type checking, automated tests, and production build pass.

### Phase 1 — Compliance MVP

Build organizations, memberships, tenant-scoped content, approval and publication,
assignments, completion rules, attestations, versioned certificates, revocation,
administrative audit events, and CSV/PDF audit packs.

Pilot with a small number of design partners before adding industry-specific
workflows.

### Phase 2 — Skill Passport

Build evidence-backed competency claims, learner sharing controls, expiring links,
credential verification, consent withdrawal, access history, disputes, and the
first scoped employer API.

### Phase 3 — Learning Genome products

Activate question-quality review, placement, prerequisite inference, cross-course
concept mapping, adaptive paths, and explanation experiments only as data quality
and sample thresholds permit.

### Phase 4 — WhatsApp pilot

Pilot one course and one assignment flow. Measure successful linking, delivery,
replies, completion, outcome parity, opt-outs, and cost per completed learner
before expanding the channel.

## Success metrics

### Evidence foundation

- Accepted submissions captured exactly once
- Events linked to valid question, concept, and content versions
- Event-to-projection reconciliation rate
- Missing, malformed, or delayed events
- Offline queue age and successful replay rate
- Number of question versions with useful sample sizes
- Time and accuracy of a projection rebuild

### Compliance

- Time from upload to assigned course
- Organization activation and pilot-to-paid conversion
- Assignment start, completion, and on-time completion
- Audit-pack export and acceptance
- Renewal, seat expansion, and churn
- Support incidents per active organization

### Skill Passport

- Learner opt-in and active sharing
- Verified claims per active learner
- Verification link and API use
- Repeat employer verification
- Consent withdrawals and disputes
- Attempts to use expired or revoked claims

### Learning Genome

- Question-review precision
- Mastery calibration error
- Placement time saved
- Reduction in unnecessary practice
- Retention and completion changes
- Learning gains from controlled explanation experiments

### WhatsApp

- Successful opt-in and account linking
- Message delivery and learner reply rates
- Lesson start and completion
- Cross-channel resume success
- Learning-outcome parity
- Messaging cost per completed learner
- Opt-out and complaint rates

## Privacy, security, bias, and validity

Required controls:

- Tenant isolation at the data-access layer
- Least-privilege role checks on every organization operation
- Encryption in transit and at rest
- Scoped, short-lived verification tokens
- Append-only administrative audit events
- Credential revocation
- Export, correction, retention, and deletion workflows
- Pseudonymized analytics
- Explicit permission for cross-course or cross-tenant aggregation
- No training on private documents or answers without permission
- Rate limiting, abuse detection, backups, and restore tests
- Automated cross-tenant access tests

Validity controls:

- Show confidence and evidence volume with mastery
- Measure calibration differences across sufficiently represented cohorts
- Avoid penalizing low bandwidth, device limitations, language, or channel
- Provide appeal and correction routes
- Revalidate claims after material content, scoring, or taxonomy changes
- Do not describe observational explanation correlations as causal effects

## Explicit non-goals

BookQuest will not initially:

- make automated hiring decisions or candidate rankings;
- present mastery as a guarantee of job performance;
- use blockchain credentials;
- add biometric or invasive remote proctoring;
- build a global concept graph before evidence supports it;
- mix proprietary tenant data into a global model without permission;
- promise universal legal compliance;
- replace the application with premature microservices;
- implement advanced psychometrics before the ledger is reliable; or
- launch every possible WhatsApp flow before validating one focused pilot.

## Current implementation status

The repository now contains the first Phase 0 implementation slice:

- append-only `learning_events` with idempotent event IDs;
- separate pseudonymous learner identities;
- stable concept, question, and question-version records;
- course versions and generator provenance fields;
- server-issued lesson, review, and persisted fresh/standard practice sessions;
- server-authoritative grading for lesson, practice, and review answers;
- semantic answer uniqueness and transactional mastery/source projections;
- strict evidence-reconciled, idempotent lesson completion;
- an account-scoped browser answer outbox for interrupted requests; and
- learning-ledger health counters in the admin view.

Remaining Phase 0 work is tracked in the technical ledger document and should be
completed before the compliance pilot.
